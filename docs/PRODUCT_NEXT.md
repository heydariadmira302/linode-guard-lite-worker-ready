# Linode Guard Lite 下一阶段产品方案

> 状态：需求对齐稿。先按本文定方向，再进入开发。

## 1. 核心开发前提

### 1.1 API / Service 优先

本项目必须继续坚持 API-first / Service-first：

- Telegram 只是展示窗口和交互入口。
- 业务逻辑必须沉到 `src/services/*`、`src/storage/*` 和 HTTP API 层。
- Telegram handler / callback 只负责：
  - 展示中文菜单
  - 收集用户输入
  - 调用 service / API
  - 渲染结果
- 不要把新增业务逻辑直接堆在 Telegram callback 文件里。
- 后续 Web 后台、CLI、Cloudflare Worker API 都应该复用同一套 service 逻辑。

### 1.2 功能拆文件

不要把所有内容挤在一个大文件里。

推荐按功能拆分：

```text
src/services/group-service.ts
src/services/account-service.ts
src/services/instance-service.ts
src/services/security-service.ts
src/services/admin-presence-service.ts
src/services/schedule-service.ts

src/api/groups.ts
src/api/accounts.ts
src/api/instances.ts
src/api/security.ts
src/api/admin-presence.ts
src/api/schedules.ts

src/telegram/group-renderer.ts
src/telegram/account-flow.ts
src/telegram/instance-renderer.ts
src/telegram/security-renderer.ts
src/telegram/admin-presence-renderer.ts
src/telegram/schedule-renderer.ts
src/telegram/keyboards.ts
```

原则：一个文件只负责一个清晰功能。

### 1.3 中文优先

Telegram 里的按钮、菜单、提示、错误原因必须尽量中文描述。

不要出现用户看不懂的英文按钮，例如：

```text
boot
shutdown
delete
checkin
schedule
```

应该显示为：

```text
开机
关机
删除
打卡
定时任务
```

内部 callback_data 可以继续用英文机器码，但用户可见文本必须中文。

## 2. 已确认产品规则

### 2.1 分组规则

- 一个账号只属于一个分组。
- 一个分组可以包含多个账号。
- 默认分组为：`未分组`。
- 分组下可以查看账号和服务器。
- 分组后续要支持批量操作。

### 2.2 安全事件基线

添加账号时必须建立安全基线：

- 历史登录记录不通知。
- 从添加 key 之后的新登录开始通知。
- 添加账号时记录：
  - `security_baseline_at`
  - 当前最新 login id / cursor（如 Linode API 可用）

### 2.3 Telegram 键盘规则

固定 Reply Keyboard：

```text
🏠 主菜单 | ❤️ 打卡
🖥 服务器 | 👤 账号
```

Inline Keyboard 用于当前页面操作，例如：

```text
详情
开机
关机
删除
确认
取消
下一页
返回
```

### 2.4 保活策略规则

- 保活策略以用户设置为准。
- 已配置并启用的保活策略，到触发时间后会由 Cron 执行，不再在执行时等待人工确认。
- 因此，创建或编辑高危策略时必须把确认前置：`delete_all_instances` 必须给出明显高危提示，但按当前产品规则不要求额外文本二次确认。
- 编辑已有策略并把动作改为 `delete_all_instances` 时，也必须展示同样强警告。
- 保活策略触发的自动批量删机优先级最高，但必须严格按 scope 执行。
- 其他高危功能需要二次确认。

### 2.5 定时任务实例范围规则

- 定时任务 action 只支持 `boot` / `shutdown` / `reboot`，不支持 `delete`。
- 定时任务 scope 支持 `all` / `account` / `group` / `instance`。
- `account` scope 创建时必须校验账号存在且 active。
- `group` scope 创建时必须校验分组存在。
- `instance` scope 创建时必须同时提供 `account_id` 和 `instance_id`，并校验该实例属于该账号。
- 如果 instance scope 找不到目标，不能创建未来会空跑的任务；执行阶段也不能把 `success total=0` 当成成功。

### 2.6 Protected instance 建议

如果启用批量删除或保活自动删机，建议将 protected instance 作为上线前或上线后第一优先级：

- 关键实例可标记为 protected。
- 单实例删除、批量删除、保活自动删机都必须统一检查 protected 状态。
- 自动删除跳过 protected 实例时，应在 Telegram 和审计日志中明确说明。

### 2.7 服务器 IP 展示

服务器列表和详情需要展示 IPv4。

- 列表显示第一个 IPv4。
- 详情显示全部 IPv4。
- IPv6 暂不展示。

## 3. Telegram 主菜单目标

主菜单当前建议保持高频入口和低频/高危入口分层：

```text
🏠 主菜单

🖥 服务器 | 👤 账号
❤️ 打卡 | ⏰ 定时
🛡 安全 | 📋 更多
```

更多功能：

```text
⚡ 批量 | 📁 分组
📄 审计 | 🔒 隐私
⚙️ 设置
🏠 主菜单
```

固定 Reply Keyboard：

```text
🏠 主菜单 | ❤️ 打卡
🖥 服务器 | 👤 账号
```

每条消息底部尽量提供常用快捷按钮：

```text
❤️ 打卡
🏠 主菜单
```

隐私清理：

- 更多功能里提供「🔒 隐私」入口。
- 设置页也提供「🔒 隐私清理」入口。
- 支持按钮设置 Telegram 消息自动删除：关闭 / 1 分钟 / 5 分钟 / 15 分钟 / 1 小时 / 24 小时。
- 支持「立即清理一次」，用于手动删除当前已记录、可删除的 Telegram 消息。
- 系统只记录 `chat_id` 和 `message_id` 用于删除，不保存消息正文。
- 删除由 `message_cleanup` 后台任务执行；如需接近 1 分钟清理，Cloudflare Cron 应配置为每分钟触发，Job Runner 通过 `jobs.next_run_at` 控制其他任务仍按自身节奏执行。
- Telegram Bot API 删除受平台限制：通常只能删除 48 小时内的消息；群组里需要 Bot 有删消息权限。

## 4. 账号管理目标

### 4.1 添加账号流程

当前确认流程：

```text
账号管理 -> 添加账号
1. 输入账号昵称
2. 输入 Linode Token
3. 自动检测 Token 是否正确可用
4. 未指定分组时默认进入「未分组」
5. 建立安全基线
6. 添加成功通知
```

从分组详情点击「添加账号到本组」时，流程同样是输入昵称和 Token，但新 key 直接归入当前分组；添加成功页必须提供「继续添加到本组」和「返回分组详情」，不能丢失分组上下文。

添加账号输入态必须有可视化退出路径：昵称阶段提供「取消添加 / 返回账号管理或分组详情」；Token 阶段提供「重新输入昵称 / 取消添加 / 返回账号管理或分组详情」。更新 Token 流程也必须提供「取消更新 / 返回账号详情」。

### 4.2 账号昵称

账号昵称必须支持中文，例如：

```text
西班牙1
日本备用
洛杉矶主号
```

建议规则：

- 长度 1-32。
- 支持中文、英文、数字、空格、下划线、短横线。
- 禁止空昵称。
- 禁止重复昵称。
- 内部逻辑不要依赖昵称作为唯一稳定 ID。

### 4.3 Token 检测

添加账号时必须检测 key 是否正确能用。

建议检测：

```http
GET /account
GET /linode/instances
```

失败提示必须中文化：

```text
Token 无效：请检查是否复制完整。
权限不足：请确认 Token 有读取账号和实例权限。
Linode API 暂时不可用：请稍后重试。
```

成功提示：

```text
✅ 账号添加成功

账号：西班牙1
分组：西班牙
Token 状态：可用
服务器数量：3
安全基线：已建立，历史登录不通知
```

## 6. 服务器功能目标

### 6.1 服务器入口

点击“服务器”不能是空壳，必须可操作。

菜单：

```text
🖥 服务器管理

请选择查看范围：

全部服务器
按分组查看
按账号查看
运行中
已关机
```

### 6.2 服务器列表

列表展示：

```text
🟢 web-01
账号：西班牙1
分组：西班牙
地区：es-mad
IPv4：203.0.113.10
状态：运行中
```

### 6.3 服务器详情

详情展示：

```text
🖥 服务器详情

名称：web-01
ID：123456
账号：西班牙1
分组：西班牙
地区：es-mad
状态：运行中
IPv4：
- 203.0.113.10
- 203.0.113.11
```

IPv6 暂不展示。

### 6.4 服务器详情 / 操作体验目标

服务器详情页应从“按钮集合”升级为“状态卡片 + 当前主操作 + 更多管理”。

核心信息：

```text
名称 / ID / 账号 / 分组 / 地区 / IPv4 / 状态 / 保护状态 / Bot 管理状态
```

根据状态展示一层主操作：

- 运行中：关机、重启、管理更多、返回上一列表、服务器管理
- 已关机：开机、管理更多、返回上一列表、服务器管理
- 处理中 / 未知：刷新状态、管理更多、返回上一列表、服务器管理

删除不应和开机/关机/重启平级出现。推荐路径：

```text
服务器详情 -> 管理更多 -> 危险操作 -> 删除服务器 -> 确认删除
```

管理更多建议包含：

```text
保护设置
定时任务
危险操作
返回详情
```

删除服务器必须二次确认；重启也建议确认；关机已确认；开机可直接执行。返回按钮必须保留来源上下文，例如全部列表、运行中列表、账号服务器列表或分组服务器列表。

## 7. 安全事件目标

### 7.1 历史事件处理

添加 key 以前的历史登录不通知。

实现要求：

- 添加账号成功时建立安全基线。
- 后续只通知基线之后的新登录。

### 7.2 安全事件通知

示例：

```text
⚠️ 新登录事件

账号：西班牙1
用户：root
IP：203.0.113.10
时间：2026-05-15 17:20

这是你本人操作吗？
```

按钮：

```text
是我
不是我
❤️ 打卡
```

## 8. 定时任务目标

Telegram 内必须能按钮化新增、查看、暂停、删除定时任务。

菜单：

```text
⏰ 定时任务

新增任务
任务列表
暂停全部
返回主菜单
```

新增任务目标流程：

```text
选择动作：开机 / 关机 / 重启
选择范围：全部账号 / 分组 / 账号 / 单台服务器
选择时间：每天 08:50 / 每天 23:05 / 自定义 Cron
保存
```

当前已落地 Telegram 快速新增：支持开机 / 关机 / 重启，范围支持全部账号、单账号、分组和单台服务器，支持每天 08:50 / 每天 23:05，也支持输入 `09:30` 这类每天固定时间或 5 段自定义 Cron。定时任务由 Cron 实际执行后，必须主动通知管理员，展示任务、动作、范围、总数、成功、失败和跳过保护；通知失败不能影响任务执行或导致重复执行。

## 9. 保活打卡目标

### 9.1 固定打卡入口

Reply Keyboard 固定提供：

```text
❤️ 打卡
```

每条重要消息下方也尽量提供 Inline 打卡按钮。

### 9.2 保活策略配置

菜单：

```text
❤️ 保活打卡

当前状态：
上次打卡：2 小时前
提醒时间：12 小时
最终动作时间：24 小时
最终动作：删除所有服务器
作用范围：全部账号 / 单账号 / 分组

立即打卡
设置提醒时间
设置最终动作时间
设置最终动作
设置作用范围（全部账号 / 单账号 / 分组）
暂停保活策略
```

### 9.3 自动批量删机规则

- 用户设置了保活自动删机策略后，到触发时间自动执行。
- 这个自动动作不再二次确认。
- 设置策略时必须明确警告风险。
- 执行后必须写审计日志并主动通知管理员，通知中展示策略、动作、范围、距离上次打卡时间、总数、成功、失败、跳过保护和失败详情。
- 最终动作必须按策略作用范围执行：全部账号、单账号或分组；不能因为策略触发而误操作范围外账号。
- 通知失败不能影响已经执行的 Linode 操作，也不能导致 Cron 重复执行。

## 10. 批量操作体验目标

批量操作不应把所有范围和动作平铺。当前确认方向：

```text
批量操作
- 单账号批量操作
- 分组批量操作
- 全部账号批量操作
- 🚨 批量删除
```

普通批量流程：

```text
选择范围 -> 选择开机/关机 -> 确认 -> 执行结果
```

批量删除单独入口：

```text
🚨 批量删除 -> 选择范围 -> 风险预览 -> ⚠️ 我知道风险，继续 -> 输入 DELETE -> 执行结果
```

## 11. Telegram UI 产品原则

- 常用动作一层到达。
- 高危动作逐层进入。
- 返回路径永远清楚。
- 上下文不能丢，例如从分组添加账号后继续添加仍应留在该分组。
- 按钮排版统一：详情页放当前主操作，更多/危险/设置类功能进入二级页。
- Telegram handler 只做展示和调用 Service，业务逻辑仍在 service/API/storage 层。

## 12. 开发优先级建议

### Phase 1：Telegram 可用性和账号体验

- Reply Keyboard 固定入口。
- 主菜单中文化和美化。
- 服务器按钮真正可用，显示 IPv4。
- 添加账号支持中文昵称。
- 添加账号检测 Token。
- 添加账号成功提示。
- 添加账号时建立安全基线。

### Phase 2：分组

- 新增 groups 表。
- 账号归属分组。
- 分组菜单。
- 按分组查看账号/服务器。
- 移动账号分组。

### Phase 3：安全事件优化

- 基线之后才通知。
- 安全事件菜单中文化。
- 是我 / 不是我 / 打卡按钮。

### Phase 4：定时任务和保活策略按钮化

- Telegram 新增/查看/暂停/删除定时任务，并以卡片式摘要展示任务数、启用数、动作、范围和下次运行。
- 保活策略完整配置，并以卡片式摘要展示策略总数、启用数、高危删机数和触发流程。
- 自动删机策略按用户设置执行。

## 13. Windows 创建目标

Windows 创建继续坚持 API-first / Service-first。Telegram 创建入口先选版本，并允许选择自动生成强密码或自己输入密码；Windows 11 Enterprise LTSC 2024 为实验路线，支持 zh-cn/en-us，ISO 由 Service 自动解析，不允许用户手动输入 ISO URL。已有 VPS 使用 bin456789/reinstall 重装 Win11 作为后续高级模式保留，本轮不开发。
